1. Introducere

1.1 Scop

Scopul proiectului este dezvoltarea unei aplicatii web care permite cautarea, vizionarea si recomandarea de videoclipuri YouTube in functie de preferintele utilizatorului, istoricul interactiunilor, continutul apreciat si activitatea prietenilor.

Documentul descrie functionalitatile principale, interactiunea cu utilizatorul, constrangerile tehnice si cerintele pe care aplicatia trebuie sa le indeplineasca.

1.2 Public tinta

Aplicatia este destinata utilizatorilor care doresc sa descopere rapid continut video relevant, fara sa se bazeze exclusiv pe recomandarile generale oferite de platforme externe. De asemenea, proiectul include o zona de administrare pentru gestionarea conturilor.

1.3 Definitii si abrevieri

SIV
Similar Video Recommender, numele interfetei publice a aplicatiei.
RSS
Format XML folosit pentru exportul fluxurilor de recomandari.
Token
Sir generat la autentificare si folosit pentru validarea sesiunii.
DTO
Data Transfer Object; in proiect este folosit pentru reprezentarea videoclipurilor.

2. Descriere generala

2.1 Perspectiva produsului

Video Recommending System este o aplicatie web impartita in doua zone: frontend static, realizat cu HTML, CSS si JavaScript, si backend PHP, care expune endpoint-uri HTTP pentru autentificare, cautare, recomandari, prieteni, administrare si contul utilizatorului.

Datele persistente sunt stocate intr-o baza de date SQLite. Aplicatia foloseste YouTube Data API pentru cautarea si preluarea detaliilor despre videoclipuri.

2.2 Functionalitati generale

  • inregistrare si autentificare utilizatori;
  • cautare videoclipuri YouTube cu filtre;
  • vizionare videoclipuri si recomandari similare;
  • marcare videoclipuri ca apreciate;
  • generare recomandari personalizate;
  • recomandari bazate pe tara si preferinte;
  • recomandari bazate pe activitatea prietenilor;
  • gestionare cereri de prietenie;
  • export RSS pentru fluxuri de recomandari;
  • pagina de cont pentru utilizator;
  • panou de administrare pentru conturi de utilizator.

2.3 Clase de utilizatori

Tip utilizator Descriere Drepturi principale
Vizitator Utilizator neautentificat. Poate vedea pagina de prezentare si poate accesa login/register.
Utilizator autentificat Cont normal din tabela users. Poate cauta, viziona, aprecia videoclipuri, gestiona prieteni si contul propriu.
Administrator Cont din tabela admins. Poate schimba parolele utilizatorilor si poate sterge conturi.

2.4 Mediu de operare

  • Frontend: server local pe http://localhost:8001.
  • Backend: server PHP pe http://localhost:8081.
  • Baza de date: SQLite, fisierul database.sqlite.
  • Browser modern cu suport pentru JavaScript si Fetch API.

3. Cerinte functionale

3.1 Autentificare si inregistrare

  • Sistemul trebuie sa permita crearea unui cont nou prin username, parola, categorii preferate, limbi, durate si tara.
  • Sistemul trebuie sa respinga username-urile deja existente.
  • Sistemul trebuie sa salveze parolele sub forma de hash, nu in clar.
  • Sistemul trebuie sa autentifice utilizatorul si sa genereze un token de sesiune.
  • Dupa inregistrare sau autentificare reusita, utilizatorul trebuie sa fie directionat catre pagina de cautare.

3.2 Cautare videoclipuri

  • Utilizatorul autentificat trebuie sa poata cauta videoclipuri dupa termen text.
  • Sistemul trebuie sa permita filtrarea dupa durata, limba, perioada de publicare si ordine.
  • Rezultatele trebuie afisate sub forma de carduri cu titlu, descriere, thumbnail si actiuni.
  • Sistemul trebuie sa indice daca un videoclip este deja apreciat de utilizator.

3.3 Vizionare si recomandari similare

  • Utilizatorul trebuie sa poata deschide o pagina dedicata unui videoclip.
  • Pagina de vizionare trebuie sa afiseze playerul video, titlul, descrierea si linkul catre YouTube.
  • Sistemul trebuie sa afiseze videoclipuri similare calculate pe baza tagurilor, topicurilor, canalului, limbii, duratei si engagementului.

3.4 Aprecieri si profil de recomandare

  • Utilizatorul trebuie sa poata aprecia sau elimina aprecierea pentru un videoclip.
  • Sistemul trebuie sa actualizeze greutatile pentru tagurile si categoriile asociate videoclipului.
  • Recomandarile personalizate trebuie sa tina cont de taguri, categorii, limbi si durate preferate.
  • Sistemul trebuie sa evite recomandarea videoclipurilor deja apreciate de utilizator.

3.5 Prieteni si cereri de prietenie

  • Utilizatorul trebuie sa poata trimite o cerere de prietenie folosind username-ul celuilalt utilizator.
  • Sistemul trebuie sa previna trimiterea de cereri catre propriul cont.
  • Sistemul trebuie sa diferentieze intre o cerere existenta si o prietenie deja acceptata.
  • Destinatarul trebuie sa poata accepta sau refuza o cerere primita.
  • Utilizatorul trebuie sa poata vedea lista de prieteni si sa elimine un prieten dupa username.
  • Utilizatorul trebuie sa poata primi recomandari bazate pe videoclipurile apreciate de prieteni.

3.6 Export RSS

  • Sistemul trebuie sa permita exportul recomandarilor personalizate in format RSS.
  • Sistemul trebuie sa permita exportul recomandarilor populare in tara utilizatorului.
  • Sistemul trebuie sa permita exportul recomandarilor bazate pe activitatea prietenilor.
  • Fluxul RSS trebuie sa includa titlu, link, descriere, guid si thumbnail acolo unde este disponibil.

3.7 Contul utilizatorului

  • Utilizatorul trebuie sa isi poata vedea datele principale: username, id, data crearii, tara, limbi, durate si categorii.
  • Utilizatorul trebuie sa isi poata schimba parola prin introducerea parolei curente.
  • Daca parola curenta este gresita, sistemul trebuie sa afiseze un mesaj de eroare fara delogare.
  • Dupa schimbarea cu succes a parolei, utilizatorul trebuie sa ramana autentificat.
  • Utilizatorul trebuie sa isi poata sterge contul, confirmand actiunea prin parola curenta.

3.8 Administrare

  • Administratorul trebuie sa se autentifice separat, folosind tabela admins.
  • Sistemul trebuie sa foloseasca token-uri separate pentru administratori.
  • Administratorul trebuie sa poata vedea lista utilizatorilor.
  • Administratorul trebuie sa poata schimba parola unui utilizator.
  • Administratorul trebuie sa poata sterge conturi de utilizator si datele asociate acestora.

4. Cerinte non-functionale

4.1 Securitate

  • Parolele trebuie stocate folosind password_hash.
  • Autentificarea trebuie validata prin token-uri transmise in headerul Authorization.
  • Token-urile expirate trebuie eliminate sau considerate invalide.
  • Operatiile sensibile, precum stergerea contului si schimbarea parolei, trebuie protejate prin parola curenta.
  • Endpoint-urile de admin trebuie sa accepte doar token-uri de administrator.

4.2 Performanta

  • Numarul maxim de rezultate YouTube trebuie limitat prin configuratie.
  • Interogarile SQL trebuie sa foloseasca prepared statements.
  • Listele de recomandari trebuie limitate la un numar rezonabil de rezultate.

4.3 Utilizabilitate

  • Interfata trebuie sa fie responsive pentru desktop, tableta si mobil.
  • Mesajele de eroare trebuie sa fie explicite si sa nu ascunda cauza reala a problemei.
  • Navigatia trebuie sa ofere acces catre cautare, recomandari, prieteni, RSS si cont.

4.4 Mentenabilitate

  • Codul frontend este impartit pe pagini si fisiere JavaScript dedicate.
  • Functionalitatile reutilizabile sunt separate in fisiere utilitare.
  • Backend-ul este organizat pe domenii: auth, video, friends, admin si user.

6. Interfete externe

6.1 YouTube Data API

Aplicatia foloseste YouTube Data API pentru cautarea videoclipurilor si preluarea detaliilor despre acestea. Cheia API este definita in back-end/config.php, iar numarul maxim de rezultate este controlat prin MAX_SEARCH_RESULTS.

6.2 Interfata HTTP

Frontend-ul comunica cu backend-ul folosind fetch. Endpoint-urile raspund in general cu JSON, iar fluxurile RSS raspund cu application/rss+xml.

6.3 SQLite

Persistenta este realizata prin SQLite. Tabelele sunt initializate in db/database.php.

9. Scenarii de utilizare

9.1 Crearea unui cont

  1. Vizitatorul acceseaza pagina de register.
  2. Completeaza username, parola si preferinte.
  3. Aplicatia trimite datele catre backend.
  4. Backend-ul creeaza utilizatorul, preferintele si token-ul.
  5. Frontend-ul salveaza sesiunea si directioneaza utilizatorul catre Search.

9.2 Obtinerea recomandarilor

  1. Utilizatorul autentificat acceseaza pagina Recommendations.
  2. Frontend-ul cere recomandarile de la backend.
  3. Backend-ul citeste profilul utilizatorului si solicita rezultate YouTube.
  4. Sistemul calculeaza scorurile de similaritate si preferinta.
  5. Frontend-ul afiseaza rezultatele in sectiuni separate.

9.3 Gestionarea prietenilor

  1. Utilizatorul introduce username-ul altui utilizator.
  2. Sistemul creeaza o cerere de prietenie cu status pending.
  3. Destinatarul vede cererea in sectiunea Incoming.
  4. Destinatarul poate accepta sau refuza cererea.
  5. Dupa acceptare, cei doi utilizatori apar in listele de prieteni.

9.4 Schimbarea parolei

  1. Utilizatorul acceseaza pagina Account.
  2. Introduce parola curenta si parola noua.
  3. Daca parola curenta este gresita, sistemul afiseaza mesaj de eroare.
  4. Daca parola curenta este corecta, parola este actualizata.
  5. Utilizatorul ramane autentificat pe pagina.

10. Instalare si rulare

10.1 Cerinte

  • PHP instalat local.
  • Extensie SQLite activa in PHP.
  • Conexiune la internet pentru YouTube Data API.
  • Cheie YouTube API configurata in back-end/config.php.

10.2 Pornire locala

Aplicatia poate fi pornita cu scriptul:

./start.sh

Sau manual:

php -S localhost:8081 -t back-end back-end/router.php
php -S localhost:8001 -t front-end

10.3 URL-uri locale

  • Frontend: http://localhost:8001
  • Backend: http://localhost:8081
  • Login admin: http://localhost:8001/page/admin/login.html

11. Limitari si directii viitoare

  • Aplicatia foloseste SQLite local, potrivit pentru dezvoltare si demo, nu pentru incarcare mare.
  • Cheia YouTube API este configurata local in fisier PHP si ar trebui mutata intr-un mecanism de configurare securizat pentru productie.
  • Nu exista un sistem complet de roluri in afara separarii dintre users si admins.
  • Nu exista teste automate complete pentru toate endpoint-urile.
  • O directie viitoare este adaugarea de paginare, cautare avansata in admin si editarea preferintelor din pagina Account.
